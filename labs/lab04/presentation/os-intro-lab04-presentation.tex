\begin{frame}[fragile]{Рабочий процесс Gitflow}
\protect\phantomsection\label{ux440ux430ux431ux43eux447ux438ux439-ux43fux440ux43eux446ux435ux441ux441-gitflow}
\begin{itemize}[<+->]
\tightlist
\item
  Рабочий процесс \emph{Gitflow Workflow}. Будем описывать его с
  использованием пакета \texttt{git-flow}.
\end{itemize}

\begin{block}{Общая информация}
\protect\phantomsection\label{ux43eux431ux449ux430ux44f-ux438ux43dux444ux43eux440ux43cux430ux446ux438ux44f}
\begin{itemize}[<+->]
\tightlist
\item
  Gitflow Workflow опубликована и популяризована Винсентом Дриссеном.
\item
  Gitflow Workflow предполагает выстраивание строгой модели ветвления с
  учётом выпуска проекта.
\item
  Данная модель отлично подходит для организации рабочего процесса на
  основе релизов.
\item
  Работа по модели Gitflow включает создание отдельной ветки для
  исправлений ошибок в рабочей среде.
\item
  Последовательность действий при работе по модели Gitflow:

  \begin{itemize}[<+->]
  \tightlist
  \item
    Из ветки \texttt{master} создаётся ветка \texttt{develop}.
  \item
    Из ветки \texttt{develop} создаётся ветка \texttt{release}.
  \item
    Из ветки \texttt{develop} создаются ветки \texttt{feature}.
  \item
    Когда работа над веткой \texttt{feature} завершена, она сливается с
    веткой \texttt{develop}.
  \item
    Когда работа над веткой релиза \texttt{release} завершена, она
    сливается в ветки \texttt{develop} и \texttt{master}.
  \item
    Если в \texttt{master} обнаружена проблема, из \texttt{master}
    создаётся ветка \texttt{hotfix}.
  \item
    Когда работа над веткой исправления \texttt{hotfix} завершена, она
    сливается в ветки \texttt{develop} и \texttt{master}.
  \end{itemize}
\end{itemize}
\end{block}

\begin{block}{Процесс работы с Gitflow}
\protect\phantomsection\label{ux43fux440ux43eux446ux435ux441ux441-ux440ux430ux431ux43eux442ux44b-ux441-gitflow}
\begin{enumerate}[<+->]
\item
  Основные ветки (\texttt{master}) и ветки разработки (\texttt{develop})

  \begin{itemize}[<+->]
  \tightlist
  \item
    Для фиксации истории проекта в рамках этого процесса вместо одной
    ветки \texttt{master} используются две ветки. В ветке
    \texttt{master} хранится официальная история релиза, а ветка
    \texttt{develop} предназначена для объединения всех функций. Кроме
    того, для удобства рекомендуется присваивать всем коммитам в ветке
    \texttt{master} номер версии.
  \item
    При использовании библиотеки расширений \texttt{git-flow} нужно
    инициализировать структуру в существующем репозитории:
  \end{itemize}

\begin{verbatim}
git flow init
\end{verbatim}

  \begin{itemize}[<+->]
  \tightlist
  \item
    Для github параметр \texttt{Version\ tag\ prefix} следует установить
    в \texttt{v}.
  \item
    После этого проверьте, на какой ветке Вы находитесь:
  \end{itemize}

\begin{verbatim}
git branch
\end{verbatim}
\item
  Функциональные ветки (feature)

  \begin{itemize}[<+->]
  \tightlist
  \item
    Под каждую новую функцию должна быть отведена собственная ветка,
    которую можно отправлять в центральный репозиторий для создания
    резервной копии или совместной работы команды. Ветки
    \texttt{feature} создаются не на основе \texttt{master}, а на основе
    \texttt{develop}. Когда работа над функцией завершается,
    соответствующая ветка сливается обратно с веткой \texttt{develop}.
    Функции не следует отправлять напрямую в ветку \texttt{master}.
  \item
    Как правило, ветки \texttt{feature} создаются на основе последней
    ветки \texttt{develop}.

    \begin{enumerate}[<+->]
    \item
      Создание функциональной ветки

      \begin{itemize}[<+->]
      \tightlist
      \item
        Создадим новую функциональную ветку:
      \end{itemize}

\begin{verbatim}
git flow feature start feature_branch
\end{verbatim}

      \begin{itemize}[<+->]
      \tightlist
      \item
        Создадим новую функциональную ветку:
      \end{itemize}

\begin{verbatim}
git flow feature start feature_branch
\end{verbatim}

      \begin{itemize}[<+->]
      \tightlist
      \item
        Далее работаем как обычно.
      \end{itemize}
    \item
      Окончание работы с функциональной веткой

      \begin{itemize}[<+->]
      \tightlist
      \item
        По завершении работы над функцией следует объединить ветку
        \texttt{feature\_branch} с \texttt{develop}:
      \end{itemize}

\begin{verbatim}
git flow feature finish feature_branch
\end{verbatim}
    \item
      Ветки выпуска (release)

      \begin{itemize}[<+->]
      \tightlist
      \item
        Когда в ветке \texttt{develop} оказывается достаточно функций
        для выпуска, из ветки \texttt{develop} создаётся ветка
        \texttt{release}. Создание этой ветки запускает следующий цикл
        выпуска, и с этого момента новые функции добавить больше нельзя
        --- допускается лишь отладка, создание документации и решение
        других задач. Когда подготовка релиза завершается, ветка
        \texttt{release} сливается с \texttt{master} и ей присваивается
        номер версии. После нужно выполнить слияние с веткой
        \texttt{develop}, в которой с момента создания ветки релиза
        могли возникнуть изменения.
      \item
        Благодаря тому, что для подготовки выпусков используется
        специальная ветка, одна команда может дорабатывать текущий
        выпуск, в то время как другая команда продолжает работу над
        функциями для следующего.
      \item
        Создать новую ветку release можно с помощью следующей команды:
      \end{itemize}

\begin{verbatim}
git flow release start 1.0.0
\end{verbatim}

      \begin{itemize}[<+->]
      \tightlist
      \item
        Для завершения работы на ветке release используются следующие
        команды:
      \end{itemize}

\begin{verbatim}
git flow release finish 1.0.0
\end{verbatim}
    \item
      Ветки исправления (hotfix)

      \begin{itemize}[<+->]
      \tightlist
      \item
        Ветки поддержки или ветки \texttt{hotfix} используются для
        быстрого внесения исправлений в рабочие релизы. Они создаются от
        ветки \texttt{master}. Это единственная ветка, которая должна
        быть создана непосредственно от \texttt{master}. Как только
        исправление завершено, ветку следует объединить с
        \texttt{master} и \texttt{develop}. Ветка \texttt{master} должна
        быть помечена обновлённым номером версии.
      \item
        Наличие специальной ветки для исправления ошибок позволяет
        команде решать проблемы, не прерывая остальную часть рабочего
        процесса и не ожидая следующего цикла релиза.
      \item
        Ветку \texttt{hotfix} можно создать с помощью следующих команд:
      \end{itemize}

\begin{verbatim}
git flow hotfix start hotfix_branch
\end{verbatim}

      По завершении работы ветка \texttt{hotfix} объединяется с
      \texttt{master} и \texttt{develop}:
      \texttt{git\ flow\ hotfix\ finish\ hotfix\_branch}
    \end{enumerate}
  \end{itemize}
\end{enumerate}
\end{block}
\end{frame}

\begin{frame}[fragile]{Семантическое версионирование}
\protect\phantomsection\label{ux441ux435ux43cux430ux43dux442ux438ux447ux435ux441ux43aux43eux435-ux432ux435ux440ux441ux438ux43eux43dux438ux440ux43eux432ux430ux43dux438ux435}
Семантический подход в версионированию программного обеспечения.

\begin{block}{Краткое описание семантического версионирования}
\protect\phantomsection\label{ux43aux440ux430ux442ux43aux43eux435-ux43eux43fux438ux441ux430ux43dux438ux435-ux441ux435ux43cux430ux43dux442ux438ux447ux435ux441ux43aux43eux433ux43e-ux432ux435ux440ux441ux438ux43eux43dux438ux440ux43eux432ux430ux43dux438ux44f}
\begin{itemize}[<+->]
\item
  Семантическое версионирование описывается в манифесте семантического
  версионирования.
\item
  Кратко его можно описать следующим образом:

  \begin{itemize}[<+->]
  \tightlist
  \item
    Версия задаётся в виде кортежа
    \texttt{МАЖОРНАЯ\_ВЕРСИЯ.МИНОРНАЯ\_ВЕРСИЯ.ПАТЧ}.
  \item
    Номер версии следует увеличивать:

    \begin{itemize}[<+->]
    \tightlist
    \item
      МАЖОРНУЮ версию, когда сделаны обратно несовместимые изменения
      API.
    \item
      МИНОРНУЮ версию, когда вы добавляете новую функциональность, не
      нарушая обратной совместимости.
    \item
      ПАТЧ-версию, когда вы делаете обратно совместимые исправления.
    \end{itemize}
  \item
    Дополнительные обозначения для предрелизных и билд-метаданных
    возможны как дополнения к МАЖОРНАЯ.МИНОРНАЯ.ПАТЧ формату.
  \end{itemize}
\end{itemize}
\end{block}

\begin{block}{Программное обеспечение}
\protect\phantomsection\label{ux43fux440ux43eux433ux440ux430ux43cux43cux43dux43eux435-ux43eux431ux435ux441ux43fux435ux447ux435ux43dux438ux435}
\begin{itemize}[<+->]
\item
  Для реализации семантического версионирования создано несколько
  программных продуктов.
\item
  При этом лучше всего использовать комплексные продукты, которые
  используют информацию из коммитов системы версионирования.
\item
  Коммиты должны иметь стандартизованный вид.
\item
  В семантическое версионирование применяется вместе с
  \emph{общепринятыми коммитами}.

  \begin{enumerate}[<+->]
  \item
    Пакет Conventional Changelog

    \begin{itemize}[<+->]
    \tightlist
    \item
      Пакет Conventional Changelog является комплексным решением по
      управлению коммитами и генерации журнала изменений.
    \item
      Содержит набор утилит, которые можно использовать по-отдельности.
    \end{itemize}
  \end{enumerate}
\end{itemize}
\end{block}
\end{frame}

\begin{frame}[fragile]{Общепринятые коммиты}
\protect\phantomsection\label{ux43eux431ux449ux435ux43fux440ux438ux43dux44fux442ux44bux435-ux43aux43eux43cux43cux438ux442ux44b}
Использование спецификации Conventional Commits.

\begin{block}{Описание}
\protect\phantomsection\label{ux43eux43fux438ux441ux430ux43dux438ux435}
Спецификация Conventional Commits:

\begin{itemize}[<+->]
\item
  Соглашение о том, как нужно писать сообщения commit'ов.
\item
  Совместимо с SemVer. Даже вернее сказать, сильно связано с
  семантическим версионированием.
\item
  Регламентирует структуру и основные типы коммитов.

  \begin{enumerate}[<+->]
  \item
    Структура коммита ``` ():

    Или, по-русски:

    (): {[}необязательное тело{]} {[}необязательный нижний колонтитул{]}
    ```

    \begin{itemize}[<+->]
    \tightlist
    \item
      Заголовок является обязательным.
    \item
      Любая строка сообщения о фиксации не может быть длиннее 100
      символов.
    \item
      Тема (subject) содержит краткое описание изменения.

      \begin{itemize}[<+->]
      \tightlist
      \item
        Используйте повелительное наклонение в настоящем времени:
        «изменить» (\enquote{change} not \enquote{changed} nor
        \enquote{changes}).
      \item
        Не используйте заглавную первую букву.
      \item
        Не ставьте точку в конце.
      \end{itemize}
    \item
      Тело (body) должно включать мотивацию к изменению и
      противопоставлять это предыдущему поведению.

      \begin{itemize}[<+->]
      \tightlist
      \item
        Как и в теме, используйте повелительное наклонение в настоящем
        времени.
      \end{itemize}
    \item
      Нижний колонтитул (footer) должен содержать любую информацию о
      критических изменениях.

      \begin{itemize}[<+->]
      \tightlist
      \item
        Следует использовать для указания внешних ссылок, контекста
        коммита или другой мета информации.
      \item
        Также содержит ссылку на issue (например, на github), который
        закрывает эта фиксация.
      \item
        Критические изменения должны начинаться со слова
        \texttt{BREAKING\ CHANGE}: с пробела или двух символов новой
        строки. Затем для этого используется остальная часть сообщения
        фиксации.
      \end{itemize}
    \end{itemize}
  \item
    Типы коммитов

    \begin{enumerate}[<+->]
    \tightlist
    \item
      Базовые типы коммитов

      \begin{itemize}[<+->]
      \tightlist
      \item
        \texttt{fix:} --- коммит типа fix исправляет ошибку (bug) в
        вашем коде (он соответствует PATCH в SemVer).
      \item
        \texttt{feat:} --- коммит типа feat добавляет новую функцию
        (feature) в ваш код (он соответствует MINOR в SemVer).
      \item
        \texttt{BREAKING\ CHANGE:} --- коммит, который содержит текст
        \texttt{BREAKING\ CHANGE:} в начале своего не обязательного тела
        сообщения (body) или в подвале (footer), добавляет изменения,
        нарушающие обратную совместимость вашего API (он соответствует
        MAJOR в SemVer). BREAKING CHANGE может быть частью коммита
        любого типа.
      \item
        \texttt{revert:} --- если фиксация отменяет предыдущую фиксацию.
        Начинается с revert:, за которым следует заголовок отменённой
        фиксации. В теле должно быть написано: Это отменяет фиксацию
        (это SHA-хэш отменяемой фиксации).
      \item
        Другое: коммиты с типами, которые отличаются от \texttt{fix:} и
        \texttt{feat:}, также разрешены. Например,
        @commitlint/config-conventional (основанный на The Angular
        convention) рекомендует: chore:, docs:, style:, refactor:,
        perf:, test:, и другие.
      \end{itemize}
    \item
      Соглашения The Angular convention

      \begin{itemize}[<+->]
      \tightlist
      \item
        Одно из популярных соглашений о поддержке исходных кодов ---
        конвенция Angular (The Angular convention).
      \end{itemize}

      \begin{enumerate}[<+->]
      \tightlist
      \item
        Типы коммитов The Angular convention Конвенция Angular (The
        Angular convention) требует следующие типы коммитов:
      \end{enumerate}

      \begin{itemize}[<+->]
      \tightlist
      \item
        \texttt{build:} --- изменения, влияющие на систему сборки или
        внешние зависимости (примеры областей (scope): gulp, broccoli,
        npm).
      \item
        \texttt{ci:} --- изменения в файлах конфигурации и скриптах CI
        (примеры областей: Travis, Circle, BrowserStack, SauceLabs).
      \item
        \texttt{docs:} --- изменения только в документации.
      \item
        \texttt{feat:} --- новая функция.
      \item
        \texttt{fix:} --- исправление ошибок.
      \item
        \texttt{perf:} --- изменение кода, улучшающее
        производительность.
      \item
        \texttt{refactor:} --- Изменение кода, которое не исправляет
        ошибку и не добавляет функции (рефакторинг кода).
      \item
        \texttt{style:} --- изменения, не влияющие на смысл кода
        (пробелы, форматирование, отсутствие точек с запятой и т. д.).
      \item
        \texttt{test:} --- добавление недостающих тестов или исправление
        существующих тестов.
      \end{itemize}

      \begin{enumerate}[<+->]
      \tightlist
      \item
        Области действия (scope) Областью действия должно быть имя
        затронутого пакета npm (как его воспринимает человек, читающий
        журнал изменений, созданный из сообщений фиксации). Есть
        несколько исключений из правила «использовать имя пакета»:
      \end{enumerate}

      \begin{itemize}[<+->]
      \tightlist
      \item
        \texttt{packaging} --- используется для изменений, которые
        изменяют структуру пакета, например, изменения общедоступного
        пути.
      \item
        \texttt{changelog} --- используется для обновления примечаний к
        выпуску в CHANGELOG.md.
      \item
        отсутствует область действия --- полезно для изменений стиля,
        тестирования и рефакторинга, которые выполняются во всех пакетах
        (например, style: добавить отсутствующие точки с запятой).
      \end{itemize}
    \item
      Соглашения @commitlint/config-conventional Соглашение
      @commitlint/config-conventional входит в пакет Conventional
      Changelog. В целом в этом соглашении придерживаются соглашения
      Angular.
    \end{enumerate}
  \end{enumerate}
\end{itemize}
\end{block}
\end{frame}
